## 开发设置

### 前提条件

- **Node.js**：>= 24.0.0（用于开发）
- **pnpm**：包管理器
- **Git**：版本控制

### 克隆仓库

```bash
git clone https://github.com/nicepkg/vibe-sync.git
cd vibe-sync
```

### 安装依赖

```bash
# 如果需要，安装 pnpm
npm install -g pnpm

# 安装项目依赖
cd cli
pnpm install
```

### 项目结构

vibe-sync 使用 pnpm monorepo：

```
vibe-sync/
├── pnpm-workspace.yaml     # Workspace 配置
├── cli/                    # CLI workspace 包
│   ├── package.json        # CLI 依赖
│   ├── tsconfig.json       # TypeScript 配置
│   ├── src/                # 源代码
│   │   ├── cli/            # CLI 入口点
│   │   ├── commands/       # 命令实现
│   │   ├── adapters/       # 工具适配器
│   │   ├── core/           # 核心逻辑
│   │   ├── types/          # TypeScript 类型
│   │   └── utils/          # 工具函数
│   └── test/               # 测试（镜像 src/）
└── website/                # 文档网站
```

### 构建 CLI

```bash
cd cli
pnpm build
```

### 运行测试

```bash
# 运行所有测试
pnpm test

# 监视模式运行测试
pnpm test:watch

# 运行覆盖率测试
pnpm test:coverage

# 类型检查
pnpm typecheck

# Lint
pnpm lint
```

### 本地测试 CLI

```bash
# 链接以进行本地测试
cd cli
npm link

# 现在您可以全局运行 vibe-sync
vibe-sync --version
```

## 开发工作流

vibe-sync 遵循**测试驱动开发（TDD）**方法：

### 1. 读取当前任务

检查 `TASKS.md` 了解当前阶段和任务：

```bash
cat TASKS.md
```

### 2. 先写测试

在实现之前，编写测试：

```typescript
// cli/test/commands/my-feature.test.ts
import { describe, it, expect } from 'vitest';

describe('My Feature', () => {
  it('should do something', () => {
    // 测试实现
  });
});
```

### 3. 实现功能

编写代码使测试通过：

```typescript
// cli/src/commands/my-feature.ts
export function myFeature() {
  // 实现
}
```

### 4. 验证

```bash
# 运行测试
pnpm test

# 类型检查
pnpm typecheck

# Lint
pnpm lint
```

### 5. 标记任务完成

更新 `TASKS.md`：

```markdown
- [x] 实现我的功能
```

### 6. 提交更改

遵循 Angular 提交约定：

```bash
git add .
git commit -m "feat: add my awesome feature

实现任务 X 中描述的功能。

Co-Authored-By: Claude Sonnet 4.5 <noreply@anthropic.com>"
```

## 代码风格和标准

### TypeScript

- **严格模式**：所有代码必须通过 `strict: true`
- **禁用 any**：使用正确的类型
- **显式类型**：用于公共 API

```typescript
// ✅ 好的
export function calculateHash(content: string): string {
  return crypto.createHash('sha256').update(content).digest('hex');
}

// ❌ 不好
export function calculateHash(content: any): any {
  return crypto.createHash('sha256').update(content).digest('hex');
}
```

### 测试

- **测试覆盖率**：目标 >80%
- **单元测试**：用于单个函数
- **集成测试**：用于命令工作流
- **模拟文件系统**：使用 `mock-fs` 进行文件操作

```typescript
import { describe, it, expect, beforeEach, afterEach } from 'vitest';
import mock from 'mock-fs';

describe('My Feature', () => {
  beforeEach(() => {
    mock({
      '/fake/path': {
        'file.txt': 'content'
      }
    });
  });

  afterEach(() => {
    mock.restore();
  });

  it('should work', async () => {
    // 使用模拟文件系统测试
  });
});
```

### 提交约定

遵循 [Angular 提交约定](https://github.com/angular/angular/blob/master/CONTRIBUTING.md#-commit-message-format)：

```
<type>(<scope>): <subject>

<body>

<footer>
```

**类型**：
- `feat` - 新功能
- `fix` - Bug 修复
- `docs` - 文档更改
- `refactor` - 代码重构
- `test` - 测试更改
- `chore` - 构建/工具更改

**示例**：
```bash
git commit -m "feat(adapters): add codex TOML support"
git commit -m "fix(sync): preserve JSONC comments in OpenCode"
git commit -m "docs(readme): update installation instructions"
```

## 关键规则

### 1. 从不展开环境变量

```typescript
// ❌ 错误 - 展开变量
const config = {
  env: {
    TOKEN: process.env.GITHUB_TOKEN  // 实际值！
  }
};

// ✅ 正确 - 保留语法
const config = {
  env: {
    TOKEN: "${env:GITHUB_TOKEN}"  // 引用已保留
  }
};
```

### 2. 始终使用原子写入

```typescript
import { atomicWrite } from '@src/utils/atomic-write';

// ❌ 错误 - 不防崩溃
await fs.writeFile(path, content);

// ✅ 正确 - 原子操作
await atomicWrite(path, content);
```

### 3. 遵循 TDD 工作流

```bash
# 1. 先写测试
vim test/my-feature.test.ts

# 2. 运行测试（应该失败）
pnpm test

# 3. 实现功能
vim src/my-feature.ts

# 4. 运行测试（应该通过）
pnpm test

# 5. 提交
git commit -m "feat: add my feature"
```

### 4. 尊重配置格式精度

每个工具都有**不同**的 MCP 格式：

```typescript
// OpenCode - 独特格式！
{
  "mcp": {  // 不是 "mcpServers"！
    "server": {
      "type": "local",  // 必需！
      "environment": {  // 不是 "env"！
        "VAR": "{env:VAR}"  // 花括号！
      }
    }
  }
}
```

查看 `docs/prd.md` 第 2.2 节了解完整格式规范。

## Pull Request 流程

1. **Fork 仓库**

2. **创建功能分支**：
   ```bash
   git checkout -b feat/my-feature
   ```

3. **按照上述指南进行更改**

4. **运行测试和检查**：
   ```bash
   pnpm test
   pnpm typecheck
   pnpm lint
   ```

5. **使用常规提交进行提交**：
   ```bash
   git commit -m "feat: add my feature"
   ```

6. **推送到您的 fork**：
   ```bash
   git push origin feat/my-feature
   ```

7. **在 GitHub 上创建 Pull Request**，包含：
   - 清晰的标题和描述
   - 链接到相关 issues
   - 截图（如果有 UI 更改）
   - 测试结果

### PR 检查清单

- [ ] 测试通过（`pnpm test`）
- [ ] 类型检查通过（`pnpm typecheck`）
- [ ] Lint 通过（`pnpm lint`）
- [ ] 遵循 TDD 工作流
- [ ] 配置格式完全匹配 PRD
- [ ] 环境变量未展开
- [ ] 使用原子写入
- [ ] 常规提交消息
- [ ] 文档已更新（如果需要）

## 测试指南

### 测试什么

1. **正常路径**：正常使用场景
2. **边缘情况**：空配置、缺失文件等
3. **错误处理**：无效输入、权限错误
4. **格式转换**：正确的语法转换
5. **原子操作**：错误时回滚

### 测试结构

```typescript
describe('Feature Name', () => {
  describe('when condition A', () => {
    it('should do X', () => {
      // Arrange
      const input = setupInput();

      // Act
      const result = performAction(input);

      // Assert
      expect(result).toBe(expected);
    });
  });

  describe('when condition B', () => {
    it('should do Y', () => {
      // ...
    });
  });
});
```
